# Merge Plan: Sangpt into Sancta

## Goal

Merge the standalone `Sangpt` project into `Sancta` so that:

- `sangpt` and its `sancta_gpt` engine replace the older `sanctaGPT` role inside the Sancta agent project.
- Sancta keeps working with minimal breakage to existing imports, routes, scripts, and dashboard features.
- Sangpt CLI remains usable as a normal local learning and training interface.
- Once integrated, Sangpt learns from:
  - ongoing security monitoring,
  - project knowledge,
  - operator interactions,
  - security telemetry generated by Sancta.

The merge should preserve Sancta as the main application shell and use Sangpt as the active learning/runtime engine behind the GPT layer.

---

## Merge Strategy

Use a compatibility-first merge, not a destructive rename-first merge.

That means:

- Keep Sancta’s app structure, backend modules, dashboard, and routes as the host project.
- Bring Sangpt in as a namespaced package under Sancta rather than flattening files directly into the backend root.
- Preserve Sancta’s existing `import sancta_gpt` contract by replacing the implementation behind that import.
- Sync Sancta’s live knowledge and telemetry into Sangpt’s training/data pipeline rather than forcing Sancta to adopt Sangpt’s original folder assumptions.

This reduces breakage across:

- `siem_server.py`
- `agent_loop.py`
- `sancta.py`
- `sancta_conversational.py`
- `post_generator.py`
- training scripts
- smoke tests

## Architecture Decision

### Decision

Sancta remains the primary application.

Sangpt becomes the integrated learning/runtime subsystem for the GPT layer.

### Why

Sancta already owns:

- the external interfaces,
- backend orchestration,
- API layer,
- dashboard UX,
- security model,
- project knowledge model.

Sangpt is best used as:

- the engine behind `sancta_gpt`,
- the training and checkpoint layer,
- the persistent learning memory layer,
- the CLI training environment.

## Phase Plan

## Phase 1: Audit Before Merge

### Objectives

- inventory overlapping filenames
- inventory path assumptions
- identify private APIs already relied on by Sancta
- identify risky runtime artifacts that should not be copied blindly

### Audit checklist

- confirm where Sancta’s real runtime lives
- confirm whether the target repo is already dirty
- compare both `sancta_gpt.py` files
- find all Sancta references to:
  - `sancta_gpt`
  - `SanctaGPT`
  - `get_engine()`
  - `init()`
  - `generate_reply()`
  - `generate_post()`
  - `_checkpoint_path`
  - `_init_model()`
  - `_step`
  - `_last_loss`
  - `_docs`
- inspect Sangpt for:
  - relative path usage like `./DATA` and `./logs`
  - optional non-stdlib dependencies
  - runtime trash in source tree
  - checkpoint and DB assumptions

### Known issues discovered

- both projects define `sancta_gpt.py`
- Sangpt uses relative runtime paths
- Sangpt includes runtime/generated artifacts in its repo tree
- Sancta already depends on legacy private GPT attributes
- Sancta source repo may already be dirty, so in-place merge is risky

---

## Phase 2: Stage the Merge Safely

### Objectives

- avoid damaging the original Sancta repo
- create a clean merged workspace

### Steps

1. Copy Sancta into a new staging directory.
2. Do not merge directly into the original working tree.
3. Remove inherited `.git` metadata from the staged merged copy if the staged result should stand alone.

### Result

The merge lives in a separate folder and can be validated independently.

---

## Phase 3: Namespaced Sangpt Import

### Objectives

- avoid root-level module collisions
- preserve Sangpt internals mostly unchanged

### Steps

1. Create `backend/sangpt/`.
2. Copy Sangpt runtime modules into that folder.
3. Copy only needed data folders:
   - `DATA/conversational`
   - `DATA/security`
   - `DATA/knowledge`
   - `DATA/processed`
   - `DATA/manifests`
4. Do not copy:
   - `.venv`
   - `.venv-1`
   - `__pycache__`
   - stale logs
   - stale checkpoints
   - stale SQLite DB unless intentionally preserved

### Why

This avoids conflicts with Sancta’s backend root while keeping Sangpt as a coherent subsystem.

---

## Phase 4: Replace SanctaGPT with a Compatibility Adapter

### Objectives

- keep Sancta imports stable
- swap runtime behavior behind the old API

### Adapter responsibilities

`backend/sancta_gpt.py` should:

- expose `SanctaGPT`
- expose `get_engine()`
- expose `init()`
- expose `generate()`
- expose `generate_post()`
- expose `generate_reply()`
- expose `train_step()`
- expose `status()`

### Legacy compatibility requirements

The adapter must support old Sancta call sites that use private internals:

- `_checkpoint_path`
- `_step`
- `_last_loss`
- `_docs`
- `_params`
- `_corpus_size`
- `_initialized`
- `_init_model()`
- `_ensure_initialized()`
- `set_training_mode()`
- `set_training_doc_range()`

### Rule

Do not force a full repo-wide rename from `sancta_gpt` to `sangpt` during first integration.

That kind of rename is a second-phase cleanup task, not the safe merge path.

---

## Phase 5: Feed Sancta Data into Sangpt

### Objectives

Make Sangpt learn continuously from Sancta’s real project state.

### Input sources to sync

#### Knowledge

From:

- `knowledge/*.txt`
- `knowledge/*.md`

Into:

- `backend/sangpt/DATA/knowledge/sancta_project_knowledge.txt`

#### Security telemetry

From:

- `logs/security.jsonl`
- `logs/trust_decisions.jsonl`
- `logs/cognitive_outcomes.jsonl`

Into:

- `backend/sangpt/DATA/security/sancta_live_security.txt`

#### Operator memory

From:

- `logs/operator_memory.jsonl`

Into:

- `backend/sangpt/DATA/conversational/sancta_operator_memory.txt`

### Sync rules

- sync from Sancta into Sangpt, not the other way around
- rebuild manifest after sync
- refresh corpora before training runs
- do incremental sync periodically during runtime

### Why

This is how Sangpt “learns from ongoing security monitoring and data and knowledge in project Sancta” without forcing Sancta to rewrite its own logging system first.

---

## Phase 6: Keep Sangpt CLI Usable

### Objectives

- preserve Sangpt as a normal CLI training and experimentation interface
- make it runnable from the merged Sancta project

### Steps

1. Add `backend/run_sangpt_cli.py`.
2. Launch Sangpt’s CLI from there.
3. Add help output for `--help`.
4. Make non-interactive execution exit cleanly instead of throwing EOF errors.

### Expected behavior

- interactive shell: opens Sangpt terminal workflow
- automation/non-interactive shell: prints guidance and exits safely

---

## Phase 7: Root Import Compatibility

### Objective

Keep tests and scripts working when they import `sancta_gpt` from repo root.

### Step

Add a root shim:

`sancta_gpt.py`

Contents:

- re-export from `backend.sancta_gpt`

### Why

Pytest and ad hoc scripts may import from repo root instead of from `backend/`.

---

## Phase 8: Validation

### Required validation

#### Import checks

- `import sancta_gpt` from repo root succeeds
- `import sancta_gpt` from backend context succeeds
- `sancta_gpt.status()` reports backend as Sangpt

#### Engine checks

- `get_engine()` returns a working engine
- corpus builds successfully
- manifest regeneration works in merged paths
- checkpoint save/load works
- `generate_reply(..., use_retrieval=True)` returns output

#### Test checks

Run:

```powershell
python -m pytest backend/tests/test_sancta_gpt_smoke.py -q
```

Target:

- passing smoke tests

#### Script checks

Run:

```powershell
python backend/run_sangpt_cli.py --help
python backend/run_sancta_gpt_training.py 1
```

Target:

- no import failure
- no path failure
- no console-encoding crash

## Integration Risks

## High risk

### Module name collision

Both repos define `sancta_gpt.py`.

Mitigation:

- keep Sangpt inside `backend/sangpt/`
- make `backend/sancta_gpt.py` the adapter layer

### Private API dependency

Sancta uses old engine internals, not just public methods.

Mitigation:

- preserve expected private fields in the adapter

### Dirty host repo

Sancta may already have unrelated edits.

Mitigation:

- merge in staging, not in place

## Medium risk

### Stale manifest paths

Sangpt manifests may reference old `DATA/...` relative paths.

Mitigation:

- force manifest rebuild inside merged repo

### Runtime corpus quality

Live logs can pollute training quality.

Mitigation:

- keep sync focused
- prefer structured logs
- review telemetry sources over time

### Slow training

Sangpt is pure Python and can be slow on larger corpora.

Mitigation:

- use incremental training
- keep smoke runs small
- checkpoint often

## Low risk

### Optional LangChain bridge mismatch

Sangpt’s `sancta_bridge.py` may imply optional dependencies not required for core merge.

Mitigation:

- treat it as optional
- do not put it on the critical runtime path

---

## Rollback Plan

If the integration fails:

1. Restore the old `backend/sancta_gpt.py`.
2. Remove `backend/sangpt/`.
3. Remove `backend/run_sangpt_cli.py`.
4. Remove root `sancta_gpt.py` shim if it was added only for merged compatibility.
5. Restore prior checkpoint behavior.
6. Re-run Sancta smoke tests and server import checks.

Rollback should be file-based and reversible without needing to unwind broad renames.

## Recommended Execution Order

Use this exact order when repeating the merge later:

1. Audit both projects.
2. Stage Sancta into a separate merge directory.
3. Copy Sangpt into `backend/sangpt/`.
4. Build `backend/sancta_gpt.py` as a compatibility adapter.
5. Add `backend/sangpt/project_integration.py`.
6. Add root `sancta_gpt.py` shim.
7. Add `backend/run_sangpt_cli.py`.
8. Regenerate Sangpt manifests inside merged repo.
9. Validate smoke tests and training script.
10. Document issues and remaining risks.

---

## Success Criteria

The merge is successful when all of the following are true:

- Sancta still imports and runs with `sancta_gpt` entrypoints intact.
- Sangpt is the active backend behind those entrypoints.
- Sangpt CLI can still be launched normally.
- Sancta knowledge, security logs, and operator memory are fed into Sangpt corpora.
- smoke tests pass
- training scripts run
- no destructive changes were made to the original source repos

---

## Notes For Future Use

When repeating this merge later:

- favor compatibility over elegance in the first pass
- do not start with a large rename
- do not merge directly into a dirty host repo
- treat telemetry ingestion as a controlled sync, not a blind corpus dump
- verify old private attributes before assuming adapter work is done

This merge works best as a staged integration with a compatibility shell around Sangpt, then a cleanup/refactor phase after the system is stable.
